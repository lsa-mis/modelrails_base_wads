# UI Components

The app ships a set of accessible, on-brand UI building blocks as **ViewComponents** under the
`UI::` namespace (`app/components/ui/`). They're generated by the dev-only
[`modelrails_ui`](https://github.com/dschmura/modelrails_ui) gem, then **vendored** — committed
to this app and owned by you. Production loads only `view_component` plus these vendored files;
`modelrails_ui` is a development tool and is excluded from production.

Use them as prescribed and accessibility (WCAG 2.2 AAA) and styling come for free. Browse them
live — every state, the source, and the a11y contract — at **[`/lookbook`](/lookbook)** (development).

## Using a component

The ergonomic entry point is the `ui` helper:

```erb
<%= ui :button, "Save changes", variant: :primary %>
<%= ui :dialog, title: "Edit profile" do |d| %>
  <% d.with_trigger { tag.button("Open", type: "button") } %>
  Body content
<% end %>
```

`ui(:name, …)` renders `UI::NameComponent`, forwarding all arguments. The component class stays
an implementation detail; an unknown name raises `NameError` (a boundary guard).

### Prescribed paths (use these first)

Several components have a higher-level entry point that wires up labels, ARIA, and model logic
for you. Reach for it before `ui`:

| Need | Use | Renders |
|---|---|---|
| Form field | `f.text_field :email` / `f.text_area` / `f.file_field` (`TailwindFormBuilder`) | `UI::Input` / `Textarea` / `FileInput` **with** label, help, error, ARIA |
| Modal | `render "shared/modal", title: "…" do … end` | `UI::DialogComponent` + the `modal` Stimulus controller |
| User avatar | `avatar_for(user, size: :md)` | `UI::AvatarComponent` (handles upload / gravatar / initials / hue) |

Use `ui :input` / `ui :dialog` / `ui :avatar` directly only for standalone cases outside those
flows (a filter box outside a form, a non-user avatar, etc.).

## Adding a component

1. **Discover** — `bin/rails g modelrails_ui:list` shows the catalog and which are `installed`.
   Or browse `/lookbook`.
2. **Generate (vendor)** — `bin/rails g modelrails_ui:add badge` writes
   `app/components/ui/badge_component.rb` (and a Stimulus controller for interactive
   components, which auto-registers). You own and commit it. *(Known gaps: `form_field`,
   `qr_code`, `input_otp` don't generate cleanly.)*
3. **Use it** — `ui :badge, "New"` in new markup, or point an adapter at it to replace an
   existing seam (call sites unchanged). It already uses the app's semantic tokens.
4. **Document it** — add `spec/components/previews/ui/badge_component_preview.rb` following the
   [teaching pattern](/lookbook) (class-level overview / Use-when / Don't-use-when / a11y
   contract, per-scenario notes, a labeled `Don't`, scenarios calling `ui`). `include UIHelper`.
5. **Lock the contract** — add `spec/components/ui/badge_component_spec.rb` (markup/ARIA/parity),
   test-first.
6. **Commit** the component (+ controller + preview + spec).

## Accessibility & guardrails

- **Baked in:** AAA contrast tokens, focus rings, and 44px touch targets live *inside* the
  components — using them as prescribed keeps you compliant.
- **Boundary validation:** components validate their inputs and fail loud in development
  (e.g. an unknown button `variant:` raises), while degrading safely in production.
- **Outcome gate:** the axe-core system specs audit rendered pages at WCAG 2.2 AAA in both
  light and dark themes, so a regression fails CI regardless of how the markup was written.

## How it ships

`view_component` is a runtime dependency (the vendored components are ViewComponents loaded in
production). `modelrails_ui` lives in the `:development` group and only *generates* — it is never
loaded in production. See [Extending the template](/docs/developer/extending) for the broader philosophy.
